Official Roadmap - Bitzal Core

March 18, 2026
Version 0.1.0.0

--- ## Introduction Bitzal Core will become a user-friendly and scalable digital currency platform suitable for mass adoption. Because this is such an ambitious goal, we've created a detailed roadmap to guide development. Development will be launched in three stages. Each release will increase the processing power, ease of use, feature set, and scalability of the network. We intend to increase the network's throughput by an order of magnitude with each new development stage. Furthermore, thanks to increased treasury funding, we will be able to rapidly expand the development team. We plan to double the number of developers with each release, allowing us to accelerate the implementation of each phase. We intend to schedule Bitzal releases in a way that produces a high-quality product and provides a user-friendly experience. In addition to emphasizing quality and user experience, we will use each incremental release as a robust foundation for future development. The Bitzal Core Team believes that taking the time to create a high-quality product will generate the greatest long-term value. A rushed market launch would likely result in an inferior product that causes temporary price growth but long-term stagnation or decline. Several new technologies are required to deliver such an excellent user experience. These technologies will expand the capabilities of the backbone infrastructure supporting the network and will ensure it can handle the transaction loads and data storage requirements of a mass-market user base. Masternodes will scale and be tested using a system called "state transitions." This system will provide a mathematically predictable way of determining the quality of service that masternodes provide. Full access to the blockchain will be required to perform proper state transitions of user objects, which will reference governance objects and blockchain transactions to perform quorum operations. Masternodes that fall below an activity threshold will be automatically removed from the masternode list using a new system called "masternode blocks." ZalDrive will be implemented using IPFS — a user-friendly and popular P2P file system that will automatically handle most of the network's sharing, redundancy, and syncing operations. IPFS writes will utilize masternode quorums for security. ZalDrive is an internal system that stores user objects within the network in a decentralized way. There is no public-facing system for storing non-Bitzal related information. ZalDrive is not a replacement for consumer and enterprise applications like Dropbox; it's a mechanism to ensure smooth network functioning. Network operations will scale using a targeted model allowing for exponential growth. Hardware requirements for masternodes will gradually increase to support a larger network serving more users with greater functionality. When the first development phase is deployed, masternodes will face new hard disk requirements. They will need both a solid-state drive (for faster blockchain data retrieval) and a standard hard disk with higher storage capacity. Eventually, the network will finance the creation of specialized masternode hardware with custom requirements. Such hardware will enable massive parallelization of transaction processing, hashing, and other system functions. More powerful hardware will significantly increase network throughput. This hardware will be completely open-source and non-exploitable and will connect to commodity hardware for additional storage capacity. Over the past few months, we have been rapidly expanding our development team to accelerate development. Meanwhile, a number of core developers have been focused on delivering and implementing Bitzal 0.1.0.0 - 0.1.0.3. This update was absolutely vital, serving as a bridge between the existing Bitzal codebase and future releases. Now that 0.1.0.3 has been successfully rolled out, our entire development team is transitioning to work on new versions. To create a quality product as quickly as possible, the team had to build a solid foundation. We had to carefully plan the project roadmap, create enormous amounts of documentation, and conduct significant research to avoid time- and resource-wasting mistakes during development. We will soon open a vast amount of private documentation to the community, ensuring complete transparency of our development work. This project phase is coming to a close, with the entire team now beginning to focus on creating the actual codebase. --- ## 📅 Version 0.1.0.0 / 0.1.0.3 We determined that the first version could be significantly reduced to a manageable size, then incrementally built upon with subsequent releases.
